# なぜ Keelson か?

<sub>[← Keelson ドキュメント](../../README.md) · [English](../why-keelson.md) | 日本語</sub>

Keelson は、AI コーディングエージェントが曖昧な意図を隠れた前提に変えてしまうのを防ぎます。

このワークフローは実装前の収束に焦点を当てます。

- Requirements が WHAT と WHY を定義する。
- User Stories がユーザー操作と UI モーメントを定義する。
- Design がそれらの操作を、具体的な実装と既存コードの根拠に対応づける。
- Prototype が意図した体験が破綻なく成立するかを確認する。
- Plan が承認済みアーティファクトを実装タスクに変換する。
- 実装は Kiro ベースのループで、一度に 1 タスクずつ進む。

唯一の真実は承認済みのアーティファクト束であり、チャット履歴でも単一の生成 JSON でもありません。`audit.md` はチャット由来の継続性と次の再開ポイントを保持しますが、承認済みアーティファクトを上書きすることはありません。
