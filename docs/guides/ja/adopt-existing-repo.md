# 既存リポジトリへの導入

<sub>[← Keelson ドキュメント](../../README.md) · [English](../adopt-existing-repo.md) | 日本語</sub>

**対象読者:** Claude Code または Codex を使い、既存のコードベースに機能を追加する開発者。
**前提条件:** リポジトリに Keelson がインストール済み。
**到達点:** 実装前にエージェントを制約する、機能スコープのアーティファクトを作成します。

ブラウンフィールド作業で Keelson を使うのは、AI コーディングエージェントに、機能の形を新しく勝手に作らせるのではなく既存リポジトリを尊重させたいときです。

## ブラウンフィールドの意味

ブラウンフィールドモードは機能スコープです。Keelson を「このアプリ全体をリバースエンジニアリングする」といった包括的な主張に使うべきではありません。

1 つの機能を安全に実装できるだけのコンテキストを捉えます。

- 安全に検出できる範囲での既存の技術スタックとファイル構成
- 機能が従うべき既存の規約
- 影響範囲
- 書き込みを許可する領域
- 保護すべき領域
- 受け入れ基準
- 既知のリスクや未解決の疑問

不明な点はすべて、未解決の質問のまま残します。

## はじめ方

```bash
npx @keelson/cli install
```

そして、リポジトリ全体の書き直しではなく、機能を依頼します。

```
/keel-requirements "既存の SaaS リポジトリにチーム招待機能を追加する。既存の認証・メール・チーム設定の規約に従うこと。"
/keel-us
/keel-design
/keel-proto
/keel-plan
```

良いブラウンフィールドのプロンプトは、機能と制約を明示します。

```
/keel-requirements "既存の管理者レポート画面に CSV エクスポートを追加する。課金と認証には触れない。現在のテーブルとフィルタの規約を使う。"
```

曖昧なプロンプトは避けます。

```
/keel-requirements "このアプリをリバースエンジニアリングして"
```

これは広すぎます。実際に確認した以上を理解したふりをする仕様を誘発します。

## 確認すべきこと

Requirements と User Stories の段階で、次の判断が明示されているか確認します。

- どの機能を追加するのか?
- どのユーザーまたはロールが使えるのか?
- どの既存の画面・API・テーブル・サービスが影響を受けるのか?
- どのパスやドメインを編集してよいのか?
- どのパスやドメインが保護対象か?
- 機能が完成と言えるための受け入れ基準は何か?
- どのテストやチェックで検証するのか?

Keelson はこれを `requirements.md`・`user-stories.md`・`us-mock.html`・`design.md` に記録します。`manifest.json` はアーティファクトの状態を索引するだけです。

## 実装

承認後、`/keel-impl` は承認済みのアーティファクト束と `plan.md` を読みます。

ブラウンフィールド作業では、実装スキルは次を守る必要があります。

- 生成ファイルを `Files To Touch` の中に収める
- `Files Not To Touch` を避ける
- `design.md` に記録された明示的なリポジトリ規約に従う
- 依頼された実装が変更範囲の拡大を必要とする場合は報告する
- どの受け入れ基準がファイルやテストでカバーされたかを報告する

実装が保護領域やスコープ外の領域に触れる必要が出た場合は、まず design に戻ります。

```
/keel-design
/keel-proto
/keel-plan
```

## 検証

検証はタスクローカルです。`/keel-impl` は選択したタスクに関係するチェックを実行し、`plan.md` を更新し、残っている手動検証を報告します。

## よくある失敗

**症状:** エージェントがリポジトリの無関係な部分を変更しようとする。

**対処:** `/keel-design` で明示的な `Files Not To Touch`、またはより狭い `Files To Touch` を追加し、`/keel-proto` と `/keel-plan` を再実行する。

**症状:** Keelson が既存のスタックや規約を推測できない。

**対処:** design の質問に明示的に答えるか、関連ファイルを Keelson に指し示す。Keelson は弱い根拠からスタックを推測すべきではありません。

**症状:** 仕様が 1 機能ではなくプロダクト全体を記述してしまう。

**対処:** 機能スコープの説明で `/keel-requirements` を再実行し、後続アーティファクトを作り直す。
